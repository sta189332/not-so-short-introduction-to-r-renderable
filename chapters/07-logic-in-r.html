7.1 Opening Scene: When Code Has to Decide

Leo can now read a simple line of R as structured code.

He knows that values can be stored under names. He knows that structures hold values together. He knows that syntax gives code a readable shape.

Then he writes this:

Code 
study_hours <- 6
study_hours > 5
#> [1] TRUE

The result is not another number.

It is a judgment:

Code 
TRUE
#> [1] TRUE

Something has changed. R is no longer only calculating a value or storing an object. It is evaluating a condition.

That small judgment sits behind serious data work. Filtering rows, checking missing values, identifying learners who meet a rule, and deciding whether code should run all depend on logical thinking.

A researcher may ask:

  • Which learners studied more than five hours?
  • Which project scores are at least 80?
  • Which rows contain missing values?
  • Which learners completed their first script?
  • Which observations should be kept, removed, grouped, or inspected?

R cannot answer these questions by guessing. It needs expressions that turn questions into conditions.

NoteChapter Compass

This chapter teaches logic as R’s system of judgment. By the end, Leo should be able to:

  • recognise logical values such as TRUE and FALSE;
  • use comparison operators to ask questions;
  • use logical operators to combine or reverse judgments;
  • distinguish assignment from comparison;
  • explain why & differs from &&, and why | differs from ||;
  • understand why if needs one logical answer;
  • see how vectorised logic produces many answers;
  • use is.na() to check missing values;
  • treat wider operator families as preview material rather than a memorisation list.

7.2 What Changed After Chapter 6

Chapter 6 showed that code needs structure. A line of R is not a pile of symbols. Names, values, operators, function calls, commas, and parentheses must be arranged so R can read them.

Chapter 7 moves from structure into judgment.

The new question is not only:

Can R read this expression?

It is also:

What judgment does this expression produce?

TipCore Mental Model

Syntax tells R how to read an expression.

Logic tells R how to judge a condition.

A calculation asks, “What value should be produced?”

A logical test asks, “Is this condition true?”

That difference matters because data work constantly asks R to decide what qualifies.

7.3 Logical Values: TRUE and FALSE

R has two central logical values:

Code 
TRUE
#> [1] TRUE
FALSE
#> [1] FALSE

They are not text. They are not written inside quotation marks.

Compare:

Code 
class(TRUE)
#> [1] "logical"
class("TRUE")
#> [1] "character"

TRUE is a logical value. "TRUE" is character text.

That distinction matters because logical values can guide decisions. They can be stored, counted, reversed, combined, and used to select data.

Try a small condition:

Code 
10 > 5
#> [1] TRUE
10 < 5
#> [1] FALSE
10 == 10
#> [1] TRUE

Each expression asks a question and returns a logical answer.

The question may involve one value, as above. Later, the same idea can be applied across many values or many rows.

ImportantReliable Analysis Rule

TRUE and FALSE are real values.

They are the decision values behind filtering, checking, subsetting, and conditional execution.

7.4 From Calculation to Judgment

Arithmetic operators calculate. Comparison operators judge.

This is a calculation:

Code 
10 + 5
#> [1] 15

This is a judgment:

Code 
10 > 5
#> [1] TRUE

Both use symbols. They do different jobs.

This map is deliberately small. Leo will meet other operator families later, but these are the ones that matter most for the logic chapter.

7.5 Comparison Operators: Asking Questions

Comparison operators ask whether a relationship is true.

Use a small learner score:

Code 
score <- 82
score > 80
#> [1] TRUE
score == 82
#> [1] TRUE
score != 70
#> [1] TRUE

The first line stores a value. The remaining lines ask questions about that value.

A comparison is not just a symbol between two values. It is a question written in a form R can evaluate.

When R evaluates a condition, the result is a logical judgment. Sometimes the judgment is one value: TRUE or FALSE. Sometimes, when the same condition is applied across several values, R returns several judgments.

The figure below gives a simple map of that idea.

As shown in Figure fig-condition-to-logical-judgment, logic begins with a condition. In the first panel, one score is tested and R returns one judgment. In the second panel, the same kind of rule is applied across several scores, so R returns one judgment for each value.

This is why logical values matter. They are not decorative outputs. They are the answers R produces when code asks a condition.

7.6 Predict Before You Run

Before running the next chunk, pause and predict the output.

Will each expression return TRUE or FALSE?

Code 
score <- 82
score > 80
#> [1] TRUE
score == 80
#> [1] FALSE
score != 70
#> [1] TRUE

The point is not speed. The point is to begin reading conditions as questions.

In plain language:

  • score > 80 asks whether the score is greater than 80;
  • score == 80 asks whether the score is exactly 80;
  • score != 70 asks whether the score is not 70.

Prediction forces Leo to reason before the console answers for him.

7.7 Assignment Is Not Comparison

One of the most common beginner mistakes is confusing storage with equality.

These lines do different jobs:

Code 
score <- 80
score == 80
#> [1] TRUE

<- assigns. It stores 80 in the name score.

== compares. It asks whether score is equal to 80.

The equals sign = can assign in some contexts, and it is commonly used to name function arguments:

Code 
round(77.46, digits = 1)
#> [1] 77.5

For clarity in this book, Leo should use:

  • <- for ordinary assignment;
  • = mainly for naming function arguments;
  • == for equality tests.
WarningCommon Beginner Trap

Leo wants to test whether score is equal to 80.

He writes:

Code 
score = 80

That code runs, but it assigns. It does not test equality.

The comparison he wanted is:

Code 
score == 80
#> [1] TRUE

A single missing = changes the meaning of the line.

This is a useful moment of friction. R is not punishing the learner. It is following the operator that was written.

7.8 Logical Operators: Combining Conditions

Comparison operators create judgments. Logical operators combine or reverse those judgments.

Suppose Leo has two learner indicators:

Code 
study_hours <- 6
project_score <- 85

study_hours > 5
#> [1] TRUE
project_score > 80
#> [1] TRUE

The first comparison asks whether study hours are greater than 5. The second asks whether project score is greater than 80.

Now combine both conditions:

Code 
study_hours > 5 & project_score > 80
#> [1] TRUE

This asks whether both conditions are true.

Logical operators do not calculate new measurements. They combine or reverse logical judgments. They build rules from logical results.

  • & means AND, position by position when there are many values.
  • | means OR, position by position when there are many values.
  • ! reverses a logical value.
  • && means AND for a single control decision.
  • || means OR for a single control decision.

Try the basic forms:

Code 
study_hours <- 6
project_score <- 85
first_script_completed <- FALSE

study_hours > 5 & project_score > 80
#> [1] TRUE
study_hours > 10 | project_score > 80
#> [1] TRUE
!first_script_completed
#> [1] TRUE

AND narrows. OR widens. NOT reverses.

Together, they help R build conditions that can guide action.

7.9 & Versus &&, and | Versus ||

This is a high-risk distinction, so Leo should slow down.

R often works with many values at once. For many values, use the element-wise operators & and |.

Code 
study_hours <- c(3, 6, 8)
project_score <- c(70, 85, 90)

study_hours > 5
#> [1] FALSE  TRUE  TRUE
project_score > 80
#> [1] FALSE  TRUE  TRUE
study_hours > 5 & project_score > 80
#> [1] FALSE  TRUE  TRUE

R checks each position. The result has one logical answer for each learner.

The single-decision operators && and || are different. They are mainly useful when R needs one decision, often inside if.

Do not use them for row-by-row filtering.

Display-only example:

study_hours > 5 && project_score > 80
# Not appropriate for row-by-row logic: && uses a single decision, not one answer per value.

The same idea applies to OR:

Code 
study_hours > 5 | project_score > 80
#> [1] FALSE  TRUE  TRUE

Display-only example:

study_hours > 5 || project_score > 80
# Not appropriate for row-by-row logic: || uses a single decision, not one answer per value.
ImportantReliable Analysis Rule

Use & and | when working across vectors or dataset rows.

Use && and || when writing one control decision, usually inside if.

This distinction protects Leo from a common filtering mistake.

7.10 if: A Single Decision, Not a Filter

So far, Leo has used logic to test conditions. Now he sees logic control whether code should run.

Code 
score <- 82

if (score >= 50) {
  "pass"
}
#> [1] "pass"

The condition inside if must resolve to one logical answer: one TRUE or one FALSE.

Add an alternative path:

Code 
score <- 42

if (score >= 50) {
  "pass"
} else {
  "not yet"
}
#> [1] "not yet"

If the condition is true, R runs the first block. Otherwise, it runs the second block.

This is conditional execution.

It is not the same as filtering.

Filtering usually applies one condition across many values or many rows. if decides whether one block of code should run.

Display-only mistake:

scores <- c(45, 62, 80, 38, 91)

if (scores >= 50) {
  "pass"
}
# Error: the condition has length > 1

The problem is not the comparison itself. The comparison returns many logical values. if needs one.

TipMental Model

if asks whether code should run.

Filtering asks which values or rows should stay.

Both depend on logic, but they use logic for different jobs.

7.11 Vectorised Logic: One Rule, Many Values

R is built for working with many values at once.

Code 
scores <- c(45, 62, 80, 38, 91)
scores
#> [1] 45 62 80 38 91

Now ask one rule of all the values:

Code 
scores >= 50
#> [1] FALSE  TRUE  TRUE FALSE  TRUE

R does not test only the first value. It tests each value and returns one logical answer per score.

R applies the same rule across many values automatically.

This behaviour is called vectorised logic.

In a spreadsheet, Leo might write a formula beside each row. In R, he can write one expression that applies across a whole vector.

7.12 Predict Before You Run: One Answer or Many?

Before running the next chunk, predict the shape of the result.

Will R return one logical value or many logical values?

Code 
scores <- c(45, 62, 80, 38, 91)
scores >= 50
#> [1] FALSE  TRUE  TRUE FALSE  TRUE
scores == 80
#> [1] FALSE FALSE  TRUE FALSE FALSE
scores != 38
#> [1]  TRUE  TRUE  TRUE FALSE  TRUE

Each expression compares one vector with one rule. The result is a logical vector.

The rule is written once. R applies it to many values.

7.13 Using Logical Results to Select Values

A logical result can select matching values.

Code 
scores <- c(45, 62, 80, 38, 91)

scores >= 50
#> [1] FALSE  TRUE  TRUE FALSE  TRUE
scores[scores >= 50]
#> [1] 62 80 91

The condition inside the square brackets produces logical answers. The square brackets keep the values whose matching logical positions are TRUE.

This is the beginning of filtering.

TipMental Model

Filtering is logic applied to a data structure.

The condition decides what qualifies. The structure determines what can be selected.

Leo does not need full filtering workflows here. He only needs the principle: a condition can become a selector.

7.14 Logic with Missing Values

Real data contains missing values.

Code 
scores <- c(45, 62, NA, 38, 91)
scores
#> [1] 45 62 NA 38 91

Now compare the scores with 50:

Code 
scores >= 50
#> [1] FALSE  TRUE    NA FALSE  TRUE

R returns NA where the comparison cannot be completed.

Why?

Because R cannot decide whether a missing value is greater than or equal to 50. NA marks uncertainty. It interrupts judgment.

Leo may try this:

Code 
scores == NA
#> [1] NA NA NA NA NA

This does not work the way many beginners expect. R cannot use ordinary equality to confirm an unknown value.

To test missingness, use is.na():

Code 
is.na(scores)
#> [1] FALSE FALSE  TRUE FALSE FALSE
ImportantReliable Analysis Rule

Special values require special checks.

Use is.na() to test missingness. Do not rely on == NA.

This matters in real data cleaning. A missing value is not a blank decoration. It changes what R can know, compare, summarise, and decide.

7.15 A Small Repair Win: From Wrong Test to Reliable Check

Leo wants to find which scores are missing.

His first attempt is tempting:

Code 
scores == NA
#> [1] NA NA NA NA NA

The result is not useful for identifying the missing position.

He repairs the test:

Code 
is.na(scores)
#> [1] FALSE FALSE  TRUE FALSE FALSE

Now the missing value is visible.

This is a small win, but it is an important one. Leo has not merely memorised a function. He has learned a diagnostic rule: when a value is special, use the special check designed for it.

7.18 How R Thinks: Logic Errors Are Different from Syntax Errors

NoteDebugging Lens

When code gives the wrong logical result, Leo should ask what kind of problem he is facing:

  • syntax problem: R cannot read the expression;
  • object problem: a name does not exist;
  • type problem: the value is not the kind expected;
  • structure problem: the result has the wrong length or shape;
  • logic problem: the code runs, but the condition asks the wrong question.

A logic problem can be more dangerous than a syntax problem. A syntax problem usually stops the code. A logic problem may run and still select the wrong cases.

That is why Leo should inspect logical results before trusting them.

## Assignment Is Not Comparison

One of the most common beginner mistakes is confusing storage with equality.

These lines do different jobs:

```{r}
score <- 80
score == 80
```

`<-` assigns. It stores `80` in the name `score`.

`==` compares. It asks whether `score` is equal to `80`.

The equals sign `=` can assign in some contexts, and it is commonly used to name function arguments:

```{r}
round(77.46, digits = 1)
```

For clarity in this book, Leo should use:

- `<-` for ordinary assignment;
- `=` mainly for naming function arguments;
- `==` for equality tests.

::: {.callout-warning appearance="simple" title="Common Beginner Trap"}
Leo wants to test whether `score` is equal to 80.

He writes:

```{r}
score = 80
```

That code runs, but it assigns. It does not test equality.

The comparison he wanted is:

```{r}
score == 80
```

A single missing `=` changes the meaning of the line.
:::

This is a useful moment of friction. R is not punishing the learner. It is following the operator that was written.

## Logical Operators: Combining Conditions

Comparison operators create judgments. Logical operators combine or reverse those judgments.

Suppose Leo has two learner indicators:

```{r}
study_hours <- 6
project_score <- 85

study_hours > 5
project_score > 80
```

The first comparison asks whether study hours are greater than 5. The second asks whether project score is greater than 80.

Now combine both conditions:

```{r}
study_hours > 5 & project_score > 80
```

This asks whether both conditions are true.

Logical operators do not calculate new measurements. They combine or reverse logical judgments. They build rules from logical results.

- `&` means AND, position by position when there are many values.
- `|` means OR, position by position when there are many values.
- `!` reverses a logical value.
- `&&` means AND for a single control decision.
- `||` means OR for a single control decision.

Try the basic forms:

```{r}
study_hours <- 6
project_score <- 85
first_script_completed <- FALSE

study_hours > 5 & project_score > 80
study_hours > 10 | project_score > 80
!first_script_completed
```

AND narrows. OR widens. NOT reverses.

Together, they help R build conditions that can guide action.

## `&` Versus `&&`, and `|` Versus `||`

This is a high-risk distinction, so Leo should slow down.

R often works with many values at once. For many values, use the element-wise operators `&` and `|`.

```{r}
study_hours <- c(3, 6, 8)
project_score <- c(70, 85, 90)

study_hours > 5
project_score > 80
study_hours > 5 & project_score > 80
```

R checks each position. The result has one logical answer for each learner.

The single-decision operators `&&` and `||` are different. They are mainly useful when R needs one decision, often inside `if`.

Do not use them for row-by-row filtering.

Display-only example:

```r
study_hours > 5 && project_score > 80
# Not appropriate for row-by-row logic: && uses a single decision, not one answer per value.
```

The same idea applies to OR:

```{r}
study_hours > 5 | project_score > 80
```

Display-only example:

```r
study_hours > 5 || project_score > 80
# Not appropriate for row-by-row logic: || uses a single decision, not one answer per value.
```

::: {.callout-important appearance="simple" title="Reliable Analysis Rule"}
Use `&` and `|` when working across vectors or dataset rows.

Use `&&` and `||` when writing one control decision, usually inside `if`.
:::

This distinction protects Leo from a common filtering mistake.

## `if`: A Single Decision, Not a Filter

So far, Leo has used logic to test conditions. Now he sees logic control whether code should run.

```{r}
score <- 82

if (score >= 50) {
  "pass"
}
```

The condition inside `if` must resolve to one logical answer: one `TRUE` or one `FALSE`.

Add an alternative path:

```{r}
score <- 42

if (score >= 50) {
  "pass"
} else {
  "not yet"
}
```

If the condition is true, R runs the first block. Otherwise, it runs the second block.

This is conditional execution.

It is not the same as filtering.

Filtering usually applies one condition across many values or many rows. `if` decides whether one block of code should run.

Display-only mistake:

```r
scores <- c(45, 62, 80, 38, 91)

if (scores >= 50) {
  "pass"
}
# Error: the condition has length > 1
```

The problem is not the comparison itself. The comparison returns many logical values. `if` needs one.

::: {.callout-tip appearance="simple" title="Mental Model"}
`if` asks whether code should run.

Filtering asks which values or rows should stay.
:::

Both depend on logic, but they use logic for different jobs.

## Vectorised Logic: One Rule, Many Values

R is built for working with many values at once.

```{r}
scores <- c(45, 62, 80, 38, 91)
scores
```

Now ask one rule of all the values:

```{r}
scores >= 50
```

R does not test only the first value. It tests each value and returns one logical answer per score.

R applies the same rule across many values automatically.

This behaviour is called vectorised logic.

In a spreadsheet, Leo might write a formula beside each row. In R, he can write one expression that applies across a whole vector.

## Predict Before You Run: One Answer or Many?

Before running the next chunk, predict the shape of the result.

Will R return one logical value or many logical values?

```{r}
scores <- c(45, 62, 80, 38, 91)
scores >= 50
scores == 80
scores != 38
```

Each expression compares one vector with one rule. The result is a logical vector.

The rule is written once. R applies it to many values.

## Using Logical Results to Select Values

A logical result can select matching values.

```{r}
scores <- c(45, 62, 80, 38, 91)

scores >= 50
scores[scores >= 50]
```

The condition inside the square brackets produces logical answers. The square brackets keep the values whose matching logical positions are `TRUE`.

This is the beginning of filtering.

::: {.callout-tip appearance="simple" title="Mental Model"}
Filtering is logic applied to a data structure.

The condition decides what qualifies. The structure determines what can be selected.
:::

Leo does not need full filtering workflows here. He only needs the principle: a condition can become a selector.

## Logic with Missing Values

Real data contains missing values.

```{r}
scores <- c(45, 62, NA, 38, 91)
scores
```

Now compare the scores with 50:

```{r}
scores >= 50
```

R returns `NA` where the comparison cannot be completed.

Why?

Because R cannot decide whether a missing value is greater than or equal to 50. `NA` marks uncertainty. It interrupts judgment.

Leo may try this:

```{r}
scores == NA
```

This does not work the way many beginners expect. R cannot use ordinary equality to confirm an unknown value.

To test missingness, use `is.na()`:

```{r}
is.na(scores)
```

::: {.callout-important appearance="simple" title="Reliable Analysis Rule"}
Special values require special checks.

Use `is.na()` to test missingness. Do not rely on `== NA`.
:::

This matters in real data cleaning. A missing value is not a blank decoration. It changes what R can know, compare, summarise, and decide.

## A Small Repair Win: From Wrong Test to Reliable Check

Leo wants to find which scores are missing.

His first attempt is tempting:

```{r}
scores == NA
```

The result is not useful for identifying the missing position.

He repairs the test:

```{r}
is.na(scores)
```

Now the missing value is visible.

This is a small win, but it is an important one. Leo has not merely memorised a function. He has learned a diagnostic rule: when a value is special, use the special check designed for it.

## How R Thinks: Logic Errors Are Different from Syntax Errors

::: {.callout-note appearance="simple" title="Debugging Lens"}
When code gives the wrong logical result, Leo should ask what kind of problem he is facing:

- syntax problem: R cannot read the expression;
- object problem: a name does not exist;
- type problem: the value is not the kind expected;
- structure problem: the result has the wrong length or shape;
- logic problem: the code runs, but the condition asks the wrong question.
:::

A logic problem can be more dangerous than a syntax problem. A syntax problem usually stops the code. A logic problem may run and still select the wrong cases.

That is why Leo should inspect logical results before trusting them.

## Common Beginner Mistakes

| Mistake | Why it causes trouble | Better habit |
|---|---|---|
| Using `=` or `<-` when testing equality | assignment stores a value instead of asking a question | use `==` for equality tests |
| Putting logical values in quotes | `"TRUE"` is text, not logical `TRUE` | write `TRUE` and `FALSE` without quotes |
| Using `== NA` to find missing values | `NA` marks uncertainty and needs a special check | use `is.na()` |
| Using `&&` or `||` for row-by-row rules | they are single-decision operators | use `&` and `|` for vectors and rows |
| Giving `if` a vector condition | `if` needs one `TRUE` or `FALSE` | use `if` for one decision and filtering for many values |
| Forgetting to inspect logical output | a condition may run but ask the wrong question | print or inspect the logical result first |
: Common beginner mistakes with logic and operators {#tbl-logic-mistakes}

These mistakes are not signs that Leo cannot code. They are signs that logic has rules, and those rules become clearer with practice.
