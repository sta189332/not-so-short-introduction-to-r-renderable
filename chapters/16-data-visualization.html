16.3 Why Visualization Comes After Preparation

A plot looks immediate. R draws it quickly, sometimes with one line of code.

That speed can mislead beginners.

Visualization should not be treated as the first step in analysis. It depends on everything Leo has already learned. A plot is only as trustworthy as the data behind it.

If project_score is still stored as text, a scatter plot will fail or mislead. If completion_status contains "Completed", "completed ", "COMPLETE", and "Yes" as separate categories, a bar chart will split one idea into several bars. If missing values have not been recognised, a plot may quietly remove rows. If rows are duplicated, a count plot may exaggerate a group.

So visualization comes after importing, cleaning, transforming, and summarising because plotting does not repair weak evidence. It reveals the consequences of earlier decisions.

16.4 Data Frames, Tidy Data, and Why Plots Need Structure

In Chapter 5, Leo learned that a data frame is one of R’s most important structures for analytical work. It holds variables in columns and observations in rows.

That structure matters here because most modern plotting in R expects data to arrive as a data frame or something very close to it.

A plot needs to know:

  • which variable goes on the x-axis
  • which variable goes on the y-axis
  • which variable defines colour, fill, size, shape, or grouping
  • whether each variable is continuous or discrete
  • whether each row is one observation or one already summarised result

Chapter 14 added another important idea: tidy data. In tidy data, each variable has its own column, each observation has its own row, and each value has its own cell. That arrangement makes plotting easier because visual mappings can point directly to columns.

Each column has a job. Week can become a horizontal time axis. Study_hours can become a numeric input. Project_score can become a vertical outcome. Completion_status can become a colour or fill category.

The data frame supplies the evidence. The plot decides how the evidence becomes visible.

That is the first reason ggplot2 will feel different from older plotting habits. It does not ask Leo to pull vectors out of the data frame one by one. It asks him to keep the data together and describe how variables should be mapped into a visual form.

Some are good for quick diagnostic plots. Some are good for multi-panel statistical graphics. Some are good for interactivity. Some are good for dashboards or maps. Some are designed around a grammar of graphics.

16.7 The Grammar of Graphics: The Idea Behind ggplot2

The name ggplot2 comes from the grammar of graphics, a way of thinking about statistical graphics associated with Wilkinson (2005) and implemented for R by Wickham (2016).

A grammar matters because it gives Leo something deeper than a catalogue of chart types.

Instead of memorising one command for scatter plots, another for bar charts, another for lines, and another for histograms, Leo learns the building blocks behind them.

A plot is built from parts:

  • data supplies the evidence
  • aesthetic mappings connect variables to visual attributes
  • geometries choose the visual marks, such as points, lines, or bars
  • statistical transformations compute counts, bins, means, or summaries when needed
  • scales translate data values into positions, colours, sizes, and labels
  • facets split one plot into small panels
  • themes control non-data appearance
  • coordinates control the plotting space
  • layers are drawn in order

This is the lightbulb moment.

Leo is not asking R to “make a chart” in one mysterious act. He is building a visual argument from understandable parts.

16.13 Histograms: Bins and Distributions

The word bins matters. A histogram does not show every value as a separate category. It divides the numeric range into intervals and counts how many observations fall into each interval.

16.15 Continuous and Discrete Variables

Plots behave differently depending on the type of variable.

A continuous variable has numeric values across a range. Examples include study_hours, project_score, confidence_score, lessons_completed, and payment_probability_score.

A discrete variable has categories. Examples include background, completion_status, preferred_learning_mode, region, and r_experience_level.

This matters because ggplot2 chooses scales based on variable type.

If Leo maps project_score to the y-axis, ggplot2 creates a continuous scale. If he maps background to the x-axis, ggplot2 creates a discrete scale. If he maps completion_status to colour, ggplot2 creates a discrete colour legend. If he maps payment_probability_score to colour, ggplot2 creates a continuous colour scale.

The plot is not only reading names. It is reading types.

That links this chapter back to Chapter 4, where Leo learned that values have types. Two values can look similar on the screen and still behave differently inside R. In visualization, that difference becomes visible.

WarningCheck the type before blaming the plot

If a numeric variable behaves like categories, it may have been imported as text. If a category behaves like a numeric scale, it may need to be converted or recoded. Plotting problems often point back to data type problems.

16.17 Statistical Transformations: Counts, Bins, Means, and Summaries

Some geoms draw the data exactly as supplied. geom_point() draws points from existing x and y values.

This is why ggplot2 often uses the language of geoms and stats.

## The Grammar of Graphics: The Idea Behind ggplot2

The name `ggplot2` comes from the grammar of graphics, a way of thinking about statistical graphics associated with Wilkinson (2005) and implemented for R by Wickham (2016).

A grammar matters because it gives Leo something deeper than a catalogue of chart types.

Instead of memorising one command for scatter plots, another for bar charts, another for lines, and another for histograms, Leo learns the building blocks behind them.

A plot is built from parts:

* **data** supplies the evidence
* **aesthetic mappings** connect variables to visual attributes
* **geometries** choose the visual marks, such as points, lines, or bars
* **statistical transformations** compute counts, bins, means, or summaries when needed
* **scales** translate data values into positions, colours, sizes, and labels
* **facets** split one plot into small panels
* **themes** control non-data appearance
* **coordinates** control the plotting space
* **layers** are drawn in order

This is the lightbulb moment.

Leo is not asking R to "make a chart" in one mysterious act. He is building a visual argument from understandable parts.

The most basic mental model is:

```{r}
#| label: chapter16-ggplot-template
#| lst-label: lst-chapter16-ggplot-template
#| lst-cap: "Code for the Basic ggplot2 Template"
#| eval: false

ggplot(data = ...) +
  geom_...(mapping = aes(...))
```

The code in @lst-chapter16-ggplot-template gives Leo the basic grammar-of-graphics template.

Read that slowly.

`ggplot(data = ...)` starts with the data frame.

`aes(...)` says how variables in that data frame should become visual properties.

`geom_...()` says what kind of mark should be drawn.

The `+` sign adds parts to the plot.

::: {.callout-important appearance="simple" title="The ggplot2 sentence"}
A `ggplot2` plot reads like a sentence: "Use this data, map these variables, draw this kind of mark, then add more instructions if needed."

## Mapping Versus Setting

One of the most common beginner mistakes in `ggplot2` is confusing mapping with setting.

Mapping connects a variable to a visual attribute.

Setting gives a visual attribute one fixed value.

These two lines look similar, but they mean very different things:

```{r}
#| label: chapter16-mapping-setting
#| lst-label: lst-chapter16-mapping-setting
#| lst-cap: "Code for Comparing Mapping and Setting"
#| eval: false

aes(color = completion_status)

color = "steelblue"
```

The code in @lst-chapter16-mapping-setting contrasts mapping and setting.

The first line maps a variable. It tells `ggplot2` to use colour to represent different completion statuses.

The second line sets a constant. It tells `ggplot2` to make the points steelblue, regardless of what the data say.

Leo can see the difference in code.

```{r}

#| lst-label: lst-fig-16-3-mapping-completion-status

  data = df_plot,

```

The code in @lst-fig-16-3-mapping-completion-status produces @fig-16-3-mapping-completion-status.

If the visual property should change according to a variable in the data, put it inside `aes()`. If the visual property should stay the same for every mark, put it outside `aes()`.

## Continuous and Discrete Variables

Plots behave differently depending on the type of variable.

A continuous variable has numeric values across a range. Examples include `study_hours`, `project_score`, `confidence_score`, `lessons_completed`, and `payment_probability_score`.

A discrete variable has categories. Examples include `background`, `completion_status`, `preferred_learning_mode`, `region`, and `r_experience_level`.

This matters because `ggplot2` chooses scales based on variable type.

If Leo maps `project_score` to the y-axis, `ggplot2` creates a continuous scale. If he maps `background` to the x-axis, `ggplot2` creates a discrete scale. If he maps `completion_status` to colour, `ggplot2` creates a discrete colour legend. If he maps `payment_probability_score` to colour, `ggplot2` creates a continuous colour scale.

The plot is not only reading names. It is reading types.

That links this chapter back to Chapter 4, where Leo learned that values have types. Two values can look similar on the screen and still behave differently inside R. In visualization, that difference becomes visible.

::: {.callout-warning appearance="simple" title="Check the type before blaming the plot"}
If a numeric variable behaves like categories, it may have been imported as text. If a category behaves like a numeric scale, it may need to be converted or recoded. Plotting problems often point back to data type problems.
:::

## Statistical Transformations: Counts, Bins, Means, and Summaries

Some geoms draw the data exactly as supplied. `geom_point()` draws points from existing x and y values.

Other geoms compute something before drawing. `geom_bar()` counts rows. `geom_histogram()` bins values and counts observations in those bins. `stat_summary()` can calculate means or other summaries before drawing.

This is why `ggplot2` often uses the language of geoms and stats.

Leo does not need advanced statistical graphics yet, but he does need to know when the plotted value already exists and when the plot is computing it.

For example, this plot calculates the mean project score inside the plot:

```{r}

#| lst-label: lst-fig-16-9-stat-summary-mean
#| lst-cap: "Code for Mean Project Score by Preferred Learning Mode Calculated with Stat_summary()"
#| fig-cap: "Mean Project Score by Preferred Learning Mode Calculated with stat_summary()."
#| fig-alt: "A plot showing mean project score by preferred learning mode using a statistical summary layer."
#| fig-width: 7
#| fig-height: 5

ggplot(
  data = df_plot,
  mapping = aes(x = preferred_learning_mode, y = project_score)
) +
  stat_summary(fun = mean, geom = "point", size = 3) +
  stat_summary(fun = mean, geom = "crossbar", width = 0.5) +
  labs(
    title = "Mean project score by preferred learning mode",
    x = "Preferred learning mode",
    y = "Mean project score"
  ) +
  theme_minimal()
```

The code in @lst-fig-16-9-stat-summary-mean produces @fig-16-9-stat-summary-mean.

This is useful for teaching what `ggplot2` can compute. However, in serious analysis, Leo should often calculate important summaries explicitly before plotting. That makes the evidence easier to inspect, test, and explain.

A good habit is:

1. use plot-level statistics for quick exploration
2. create summary tables when the calculation matters to the argument
3. plot the inspected summary table when reporting a result

That habit keeps visualization connected to reproducible analysis.

## Multi-Layer Rendering Order

A `ggplot2` plot is built in layers.

The order matters.

Layers added later are drawn on top of earlier layers. That can help Leo emphasise important marks, but it can also hide information if he is careless.

Here is a simple layered plot.

```{r}
#| label: fig-16-10-layer-order
#| lst-label: lst-fig-16-10-layer-order
#| lst-cap: "Code for A Layered Plot Where the Summary Mean Is Drawn After the Individual Points"
#| fig-cap: "A Layered Plot Where the Summary Mean Is Drawn After the Individual Points."
#| fig-alt: "A layered plot showing individual points in the background and summary means drawn on top."
#| fig-width: 7
#| fig-height: 5

ggplot(
  data = df_plot,
  mapping = aes(x = preferred_learning_mode, y = project_score)
) +
  geom_jitter(width = 0.15, alpha = 0.35) +
  stat_summary(fun = mean, geom = "point", size = 4) +
  labs(
    title = "Individual scores with group means",
    x = "Preferred learning mode",
    y = "Project score"
  ) +
  theme_minimal()
```

The code in @lst-fig-16-10-layer-order produces @fig-16-10-layer-order.

The jittered points appear first. They show individual variation. The mean points appear after them, so they are drawn on top.

If Leo reversed the order, the individual points could cover the summary marks.

That is the layer rule:

> In `ggplot2`, later layers are drawn over earlier layers.

This is not only technical. It is part of visual reasoning. Leo decides what should sit in the background and what should sit in the foreground.

## Saving and Exporting Plots with ggsave()

A plot in the RStudio Plots pane is not yet a deliverable.

Leo will often need to save a figure for a report, slide, article, or Quarto document. `ggplot2` provides `ggsave()` for this task.

The safest habit is to assign the plot to an object and then save it.

```{r}
#| label: chapter16-save-plot
#| lst-label: lst-chapter16-save-plot
#| lst-cap: "Code for Saving a Named Plot with ggsave()"
#| eval: false

completion_plot <- ggplot(
  data = df_completion_rate,
  mapping = aes(x = background, y = completion_rate)
) +
  geom_col() +
  scale_y_continuous(
    labels = function(x) paste0(round(x * 100), "%"),
    limits = c(0, 1)
  ) +
  coord_flip() +
  labs(
    title = "Completion rate by learner background",
    x = "Learner background",
    y = "Completion rate"
  ) +
  theme_minimal()

ggsave(
  filename = "outputs/figures/chapter16-completion-rate.png",
  plot = completion_plot,
  width = 7,
  height = 5,
  dpi = 300
)
```

The code in @lst-chapter16-save-plot shows how to save a named plot with explicit size and resolution.

This code names the file, names the plot object, sets dimensions, and specifies resolution.

That is much better than saving whatever happens to be in the plotting window.

::: {.callout-tip appearance="simple" title="Export plots deliberately"}
When a plot matters, save it with a clear filename, explicit width, explicit height, and an appropriate resolution. Reproducible output should not depend on the size of Leo's current RStudio window.
:::

Later, when Leo works more deeply with Quarto, many figures will be generated directly inside reports. Still, `ggsave()` remains useful when he needs standalone image files.

## A Brief Look at Interactive Visualization

Not every plot is meant for print.

Sometimes a reader needs to hover over points, zoom into a region, filter categories, or explore values dynamically. That is where interactive visualization becomes useful.

One common route is `plotly`.

```{r}
#| label: chapter16-plotly-example
#| lst-label: lst-chapter16-plotly-example
#| lst-cap: "Code for Converting a ggplot2 Plot to plotly"

library(plotly)

p <- ggplot(
  data = df_plot,
  mapping = aes(
    x = study_hours,
    y = project_score,
    color = completion_status,
    text = learner_id
  )
) +
  geom_point(alpha = 0.65) +
  labs(
    title = "Interactive study hours and project score",
    x = "Study hours",
    y = "Project score"
  ) +
  theme_minimal()

ggplotly(p)
```

The code in @lst-chapter16-plotly-example shows how a `ggplot2` object can be converted into an interactive `plotly` object.

This takes a `ggplot2` object and turns it into an interactive plot. A reader can hover, zoom, and inspect values.

Another route is Shiny, which builds interactive applications.

```{r}
#| label: chapter16-shiny-sketch
#| lst-label: lst-chapter16-shiny-sketch
#| lst-cap: "Code for a Small Shiny App Sketch"
#| eval: false

library(shiny)

ui <- fluidPage(
  selectInput(
    inputId = "mode",
    label = "Choose learning mode",
    choices = unique(df_plot$preferred_learning_mode)
  ),
  plotOutput("score_plot")
)

server <- function(input, output, session) {
  output$score_plot <- renderPlot({
    df_plot |>
      filter(preferred_learning_mode == input$mode) |>
      ggplot(aes(x = study_hours, y = project_score)) +
      geom_point() +
      theme_minimal()
  })
}

shinyApp(ui, server)
```

The code in @lst-chapter16-shiny-sketch sketches the basic structure of a small Shiny application.

This is only a glimpse. Shiny belongs to a larger topic: building applications around analysis. Leo does not need to master it here.

The key boundary is this:

* `ggplot2` builds the visual argument
* `plotly` can make plots interactive
* Shiny can build interactive tools around plots and data

For Chapter 16, the goal is still the foundation: build a correct, readable static plot first. Interactivity should not be used to rescue a confusing design.
